## 0. Golden Rules — Never Break These

1. **This system is specifically for BPKPAD financial archives.** The only valid document types are `SPP`, `SPM`, `SP2D`, and `SPJ`.
2. **The physical archive flow starts from a box.** Before inserting any document, the app must create or select a storage box in `storage_locations`.
3. **Cloud document input goes directly into `archive_documents`.** In the latest flow, the cloud table `staging_documents` is not required. Temporary drafts may still exist locally in Room, but the main cloud table is `archive_documents`.
4. **SPJ must never stand alone.** A document with `document_type = SPJ` can only be created if it has a valid `bundle_id` from `transaction_bundles`.
5. **Original documents always have exactly one physical copy.** If `copy_type = ORIGINAL`, then `copy_count` must always be `1` and must not be editable from the UI.
6. **Copy documents may have multiple copies.** If `copy_type = COPY`, then `copy_count` can be entered and updated as long as it is at least `1`.
7. **Never send local Room UUIDs to Supabase UUID columns.** Supabase UUID columns such as `id`, `storage_location_id`, and `bundle_id` must use UUIDs generated by Supabase. If the remote UUID does not exist yet, create the remote record first.
8. **The default financial classification code is `900.1.3.1`.** If the app does not send `classification_code`, Supabase must default it to `900.1.3.1`.
9. **Use JSONB metadata for dynamic and denormalized fields.** Do not create excessive new columns for details that can safely live in `metadata`.
10. **Never put business logic inside Composables.** Composables only receive state and event callbacks from ViewModels.
11. **Never expose DTOs directly to the UI.** Remote DTOs and Room Entities must always be mapped to Domain Models.
12. **Never hardcode secrets.** Supabase URL and anon key must come from `local.properties` or `BuildConfig`.

## 4. Supabase Database — Final Financial Schema

### 4.1. ENUM Types

```sql
doc_type = 'SPP' | 'SPM' | 'SP2D' | 'SPJ'
doc_status = 'AVAILABLE' | 'BORROWED' | 'DISPOSED' | 'UNVERIFIED'
doc_condition = 'GOOD' | 'DAMAGED' | 'LOST'
document_copy_type = 'ORIGINAL' | 'COPY'
```

### 4.2. Main Tables

| Table | Purpose |
|---|---|
| `archive_classifications` | Archive classification master data. The default financial classification is `900.1.3.1`. |
| `storage_locations` | Physical archive location: room, shelf, and box number. |
| `transaction_bundles` | Groups one financial disbursement flow: SPP → SPM → SP2D → SPJ. |
| `archive_documents` | Main financial archive records. |
| `document_placements` | Physical document movement and placement history. |
| `activity_logs` | Audit trail. In development mode, `actor_id` may be `null`. |

## 5. Main Data Flow

### 5.1. Create Box Flow

1. The user opens `StorageLocationScreen` or a create-box dialog.
2. The user fills in:
    - `room`
    - `shelf`
    - `box_number`
    - optional `description`
3. The app inserts the record into Supabase table `storage_locations`.
4. Supabase returns the generated UUID as `storage_location_id`.
5. This UUID is stored in ViewModel state and used when inserting archive documents.

**Important rule:** never create an `archive_documents` record without a valid `storage_location_id`.

---

### 5.2. Direct Archive Document Input Flow

1. The user selects a box/location.
2. The user opens `RapidInputScreen` or `ManualInputScreen`.
3. The user chooses `document_type`: `SPP`, `SPM`, `SP2D`, or follows the special UI rule for `SPJ`.
4. The user chooses `copy_type`:
    - `ORIGINAL`
    - `COPY`
5. If `copy_type = ORIGINAL`, the UI must:
    - set `copy_count = 1`
    - hide or disable the `copy_count` field
    - prevent the user from changing the copy count
6. If `copy_type = COPY`, the UI may:
    - show the `copy_count` field
    - validate that the value is at least `1`
7. On submit, the app inserts the document into `archive_documents`.

---

### 5.3. Transaction Bundle Flow

`transaction_bundles` is used to group documents belonging to one financial disbursement transaction.

Ideal flow:

```text
SPP -> SPM -> SP2D -> SPJ
```

Rules:
- `SPP`, `SPM`, and `SP2D` may be created without a bundle during normal manual input.
- `SPJ` must always have a `bundle_id`.
- In the Auto-Bundle feature, the app must create `transaction_bundles` first, then create `SPP`, `SPM`, `SP2D`, and `SPJ` documents using the same `bundle_id`.
- Bundle contents must remain flexible because real field conditions may not always have complete documents.

## 7. Room Local Strategy

Room may still be used for:
- archive list cache
- local drafts when the network is unavailable
- upload queue
- temporary OCR results
- temporary Excel import preview before final submit

However, in the latest cloud flow:
- cloud `staging_documents` is not required
- after the user submits final data, records are sent to `archive_documents`
- if upload fails, keep the data in Room with `PENDING_SYNC`

Never do this:

```kotlin

```

Do this instead:

```kotlin

```

## 9. Validation Rules

### 9.1. Copy Count Rule

```kotlin
fun validateCopyRule(copyType: DocumentCopyType, copyCount: Int): Int {
    return when (copyType) {
        DocumentCopyType.ORIGINAL -> 1
        DocumentCopyType.COPY -> {
            require(copyCount >= 1) { "Copy count must be at least 1" }
            copyCount
        }
    }
}
```

UI requirements:
- Disable or hide the `copy_count` field when `copy_type = ORIGINAL`.
- Automatically set `copy_count = 1` for original documents.
- Enable `copy_count` only when `copy_type = COPY`.

The database also protects this rule using:
- `CHECK` constraint
- `normalize_archive_copy_count()` trigger

---

### 9.2. SPJ Requires Bundle

```kotlin
fun validateSpjBundle(type: DocumentType, bundleId: String?) {
    if (type == DocumentType.SPJ && bundleId.isNullOrBlank()) {
        error("SPJ can only be created through a transaction bundle.")
    }
}
```

UI rule:
- Hide `SPJ` from the normal manual input dropdown, or
- Show `SPJ` only inside the bundle flow.

The database also protects this rule using:

```sql
CHECK (document_type <> 'SPJ' OR bundle_id IS NOT NULL)
```

---

### 9.3. Storage Location Required

```kotlin
fun validateStorageLocation(storageLocationId: String?) {
    require(!storageLocationId.isNullOrBlank()) {
        "Select or create a storage box first."
    }
}
```

The database protects this rule using:

```sql
storage_location_id uuid NOT NULL
```

## 13. Supabase Development Mode

Because the app does not have login yet:
- `created_by` may be `null`
- `actor_id` in `activity_logs` may be `null`
- RLS may temporarily allow `TO public`

This is development-only.

For production:
- enable Supabase Auth
- replace public RLS policies with authenticated-user policies
- fill `created_by` from `auth.uid()`

## 14. Required Error Handling

| Case | Handling |
|---|---|
| User has not selected a box | Show: `"Select or create a storage box first."` |
| Original copy count is changed | Force it back to `1`. |
| Copy count is less than `1` | Show: `"Copy count must be at least 1."` |
| SPJ without bundle | Show: `"SPJ can only be created through a transaction bundle."` |
| Supabase rejects UUID | Check the mapper. Never send local Room UUIDs. |
| Duplicate original document | Show a warning that the original document already exists. |
| Network failure | Save to Room as `PENDING_SYNC`. |
| OCR failure | Allow manual input fallback. |

## 16. Code Style Rules

- Composables must be stateless.
- ViewModels expose `StateFlow<UiState>`.
- Business validation must live in UseCases or ViewModels, not Composables.
- Repository implementations must live in `core/data/repository`.
- Mappers must live in `core/data/mapper`.
- UI strings must live in `strings.xml`.
- Any operation that can fail must return `Result<T>` or a sealed result type.
- Never hardcode Supabase URL or anon key..
